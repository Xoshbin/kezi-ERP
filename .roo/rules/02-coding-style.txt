# Coding Style & Architectural Standards

This document outlines the official coding styles, architectural patterns, and best practices for this application. Adherence to these standards is mandatory to ensure consistency, maintainability, and robustness.

## 1. Core Principles

- **Immutability is Law:** Posted financial records (invoices, bills, journal entries) can NEVER be edited or deleted. Corrections are made only through new, reversing transactions.
- **Manual Data Entry First:** The system relies on manual input. No third-party payment integrations.
- **Business Logic-Focused TDD:** All tests must focus on the core business logic (services, calculations, state changes) using Pest.
- **No Temporary Hacks:** All development, for both features and tests, must strictly adhere to the established architectural patterns (Actions, DTOs, Services), prioritize code reusability, and align with the overall system design. Solutions must be robust and scalable, without violating core accounting principles. Short-term workarounds are forbidden.
- **Single Responsibility Principle (SRP):** Each class, method, or function should have only one reason to change. Logic should be narrowly focused and encapsulated. For example, a service should orchestrate a workflow, an action should execute a single business operation, and an observer should react to a model event. This keeps the codebase modular, maintainable, and easier to test.
- **Definitive Solution Pattern:** We have established a key pattern for creating models that require pre-save calculations. The business logic for these calculations resides exclusively within a dedicated Action that accepts a DTO. Observers are reserved for side effects (e.g., updating a parent model's totals after a line item is saved). This pattern was successfully used to fix NOT NULL constraint violations when creating VendorBillLine records.
- **Explicit Context Pattern:** This pattern provides the definitive solution for any situation where a model's attribute depends on context from a parent that isn't available during its creation. The responsibility for providing context to a new model instance must be shifted from the model itself (which is contextually unaware during creation) to the calling code (the Action or Service, which is fully aware). Instead of passing raw values and hoping the model can infer the context, the calling code must explicitly create a context-aware object (like Brick\Money\Money) and pass that complete object to the creation method.
- **Lowercase Enum & Option Values:** All enum values and any option stored in the database (e.g., status, type, state) MUST be lowercase, snake_case. CamelCase, PascalCase, or uppercase values are strictly prohibited for database storage. This ensures consistency, simplifies querying, and avoids case-sensitivity issues across different database engines.
- **Architectural Consistency:** Ensure any feature, bug fix, or change is consistent with the other parts of the app. Analyze the codebase carefully and follow existing patterns.
- **Targeted Changes:** Modify only the code that needs to be changed. Avoid unnecessary refactoring to keep commit messages clean and easy to review.
- **Respect Accounting Principles:** Do not violate core accounting principles.
- **Preserve Comments:** Do not remove comments unless they are no longer relevant. Commenting is important for code maintainability.

## 2. State Management: PHP 8.1+ Backed Enums

**Rule:** All state management (e.g., `status`, `state`, `type`) MUST be implemented using PHP 8.1+ Backed Enums. The use of class constants for defining states is deprecated and should be refactored to Enums wherever possible. Enum cases MUST be `PascalCase`, while their corresponding string values MUST be `snake_case`.

**Rationale:**
- **Type Safety:** Enums provide absolute type safety.
- **Code Clarity & Discoverability:** Enums are self-documenting. `PascalCase` for cases follows official PHP examples, treating them as object members rather than traditional constants.
- **Reduced Boilerplate:** Enums can contain their own related logic.
- **Database Consistency:** Using `snake_case` for backed string values aligns with the database convention for column and table names and is enforced by the `Lowercase Enum & Option Values` core principle.

**Example:**
```php
// app/Enums/Inventory/StockLocationType.php
enum StockLocationType: string
{
    case Internal = 'internal';
    case Customer = 'customer';
    case Vendor = 'vendor';
    case InventoryAdjustment = 'inventory_adjustment';
}

// Example usage in a model:
class StockLocation extends Model
{
    protected $casts = [
        'type' => StockLocationType::class,
    ];
}
```

## 3. Layered Architecture

The application follows a strict layered architecture to separate concerns. The primary layers are: Actions, Data Transfer Objects (DTOs), Services, Observers, and Policies.

### 3.1. Actions Layer (`app/Actions/`)

**Purpose:** To encapsulate a single, specific business operation (Command Pattern). Actions are the only place where data is written or modified.

**Rules:**
- Each Action MUST have a single public `execute()` method.
- The `execute()` method MUST be wrapped in a `DB::transaction()` to ensure atomicity.
- Actions SHOULD accept a DTO for input to enforce a strict data contract.
- Actions are organized by domain (e.g., `Accounting/`, `Sales/`, `Purchases/`).
- Actions are responsible for their own validation or should call a dedicated validation service.

### 3.2. Data Transfer Objects (`app/DataTransferObjects/`)

**Purpose:** To provide type-safe, immutable data contracts for transferring data between layers, especially into Actions.

**Rules:**
- All DTOs MUST be `readonly` classes.
- All properties MUST be `public readonly` and have strict type hints.
- Data is passed exclusively through the constructor.
- DTOs contain NO business logic. They are pure data structures.
- DTOs SHOULD be organized by domain, mirroring the `Actions` structure.

### 3.3. Service Layer (`app/Services/`)

**Purpose:** To orchestrate complex business workflows that may involve multiple Actions, validation steps, or dispatching events. Services do NOT directly modify data; they delegate that responsibility to Actions.

**Rules:**
- Services contain high-level business process logic.
- Services call Actions to perform data modifications.
- Services are responsible for dispatching domain events (e.g., `InvoiceConfirmed`).
- Services can be injected into Filament Resources, Livewire Components, and other Services.

### 3.4. Observers (`app/Observers/`)

**Purpose:** To react to Eloquent model lifecycle events (`creating`, `updating`, `deleting`) to enforce system-level data integrity rules and trigger automatic side effects.

**Rules:**
- Observers are used for **System Reactions**, NOT for business rule authorization.
- **Use Cases:**
    - Enforcing non-negotiable data integrity (e.g., preventing deletion of a posted record).
    - Automatically calculating fields (e.g., line totals).
    - Maintaining the cryptographic hash chain on `JournalEntry`.
    - Creating `AuditLog` entries.

### 3.4.1. Observer Registration

**Rule:** Observers MUST be registered using the `#[ObservedBy]` attribute directly on the corresponding Eloquent model. Manual registration in `EventServiceProvider` is prohibited.

**Rationale:**
- **Colocation of Logic:** Keeps the connection between a model and its observer in one place, improving discoverability and maintainability.
- **Clarity & Explicitness:** Makes it immediately clear which observers are acting on a model just by looking at the model file.
- **Modern Syntax:** Aligns with modern Laravel practices for auto-discovery and registration.

**Example:**
```php
// app/Models/AdjustmentDocumentLine.php
namespace App\Models;

use App\Observers\AdjustmentDocumentLineObserver;
use Illuminate\Database\Eloquent\Attributes\ObservedBy;
use Illuminate\Database\Eloquent\Model;

#[ObservedBy([AdjustmentDocumentLineObserver::class])]
class AdjustmentDocumentLine extends Model
{
    // ...
}
```

### 3.5. Policies (`app/Policies/`)

**Purpose:** To handle all user **Authorization**. Policies answer the question, "Can the current user perform this action on this model?"

**Rules:**
- All authorization checks MUST be handled by a Policy.
- Policies are checked before an action is attempted, typically in the UI layer (Filament) or at the beginning of a Service method using `Gate::authorize()`.
- Do NOT place authorization logic in Observers, Services, or Actions.

### 3.6. Events & Listeners (`app/Events/`, `app/Listeners/`)

**Purpose:** To decouple different parts of the system. An action in one domain can trigger a reaction in another without them being tightly coupled.

**Rules:**
- Events are dispatched from Services after a core business process is completed (e.g., `InvoiceConfirmed`).
- Listeners subscribe to these events to perform follow-up actions (e.g., posting a journal entry, sending a notification).

## 4. Financial Calculations

**Rule:** All monetary values MUST be handled using `Brick\Money` objects. Never use floats for financial calculations.

**Rationale:** To avoid floating-point inaccuracies and ensure precision.
**Implementation:** Use the custom `MoneyCast` on all model properties that represent money. All calculations in Services and Actions must be performed using the methods provided by the `Brick\Money` library (e.g., `plus()`, `minus()`, `multipliedBy()`).

### 4.1. MoneyCast and Currency Precision

**Rule:** The `app/Casts/MoneyCast.php` class automatically handles the conversion between major and minor currency units based on the currency's defined decimal places. All numeric input (integers, floats, numeric strings) is treated as major units.

**Rationale:** To ensure that monetary values are stored with the correct precision in the database, preventing rounding errors and maintaining data integrity across different currencies.

**Implementation Examples:**

- **Iraqi Dinar (IQD - 3 decimal places):**
    - A value of `5,000,000` IQD (major units) will be stored in the database as `5,000,000,000` (minor units).
    - The `MoneyCast` multiplies the major unit value by 1000.

- **US Dollar (USD - 2 decimal places):**
    - A value of `$500` USD (major units) will be stored in the database as `50000` (minor units).
    - The `MoneyCast` multiplies the major unit value by 100.

```php
// Example usage in a model:
class Invoice extends Model
{
    protected $casts = [
        'total_amount' => MoneyCast::class,
    ];
}

// When you set the value, you use major units:
$invoice->total_amount = 5000000; // For IQD
$invoice->save(); // Stored as 5000000000

$invoice->total_amount = 500; // For USD
$invoice->save(); // Stored as 50000

### 4.2. Safe Aggregation of Money Objects

**Rule:** When calculating totals or performing any aggregation of monetary values, you MUST use the arithmetic methods provided by the `Brick\Money\Money` objects (e.g., `plus()`, `minus()`). Direct casting of the `Money` object or its `BigDecimal` amount to `float` or `int` for summation is strictly forbidden, as it will lead to a fatal `ErrorException`.

**Rationale:** The `getAmount()` method on a `Money` object returns a `Brick\Math\BigDecimal` object, not a primitive type. Attempting to cast this object to a `float` (e.g., `$total += (float)$money->getAmount()`) will cause the application to crash. Using the built-in methods ensures that all calculations are handled with arbitrary precision and type safety.

**Implementation Example:**

```php
// Correct way to sum Money objects
$total = Money::zero('USD');
foreach ($lines as $line) {
    // $line->price is a Money object from a cast
    $total = $total->plus($line->price);
}

// Incorrect way - THIS WILL CRASH
$total = 0.0;
foreach ($lines as $line) {
    $total += (float) $line->price->getAmount(); // ErrorException
}
```

## 5. Internationalization (I18n)

**Rule:** All user-facing strings, including text in the user interface, exceptions, notifications, and any other part of the application visible to the end-user, MUST be translatable using Laravel's localization features.

**Rationale:** To ensure the application can be easily adapted for different languages and regions. This is critical for a market like Iraq, which has multiple official languages (English and Kurdish). A fully localized application provides a better user experience and broader accessibility.

**Implementation:**
- **PHP Code:** Use the `__()` helper function to retrieve translation strings.
  - Example: `throw new \Exception(__('Your invoice has been successfully posted.'));`
- **Blade Templates:** Use the `@lang()` directive for translations within views.
  - Example: `<span>@lang('messages.welcome')</span>`
- **Translation Files:** All translation strings MUST be stored in language files within the `resources/lang/` directory, following this structure:
  ```
  resources/lang/
  ├── en/
  │   └── messages.php
  └── ku/
      └── messages.php
  ```
- **No Hardcoded Strings:** Under no circumstances should user-facing text be hardcoded directly in PHP files or Blade templates.
